# OpsFlow AI — Project Constitution

OpsFlow AI is a **clean-room, synthetic-only** operational data platform built as a
portfolio project. These rules are binding for all work in this repository.

## 1. Clean-room rule
This project is a from-scratch rebuild based on architectural knowledge only.
Never access, reference, import, copy, or depend on any prior work project, work PC,
VM, company environment, or private system.

## 2. Synthetic-only rule
All data in this repo is generated by `opsflow generate-events`. No real company data,
logs, screenshots, config files, Grafana JSON, or scheduler scripts — ever.

## 3. No real identifiers or secrets
No real company names, site names, hostnames, IPs, usernames, table names, filesystem
paths, or credentials. Use generic synthetic IDs (`LOC_A01`, `OCR_GATE_02`, `ROUTER_01`).
If a detail might be sensitive, replace it with a generic synthetic name.

## 4. P0 before P1
The file-based flow (generate-events → detect-anomalies → diagnose) must work and be
tested before any Postgres/dbt work. If P1 blocks, preserve P0 as the fallback MVP.

## 5. Test, then commit
Run `pytest` after each milestone. Commit only when the repo is in a runnable,
non-broken state. Update PROGRESS.md with each milestone.

## 6. No over-engineering
No Kubernetes, Kafka, Terraform, Prometheus, LocalStack, paid LLM APIs, complex ML,
or meta "project factory" systems. Grafana and GitHub Actions are stretch-only (P3).

## 7. Deterministic RCA wording
The RCA "agent" is a **deterministic, tool-style diagnostic workflow** — rule-based
functions over real evidence in the data. Never describe it as an LLM-powered agent.
Reports must cite computed evidence, never invented explanations.

## Development notes
- Python 3.10+, `pip install -e ".[dev]"` into `.venv`
- CLI entry: `python -m opsflow <command>` (see README quickstart)
- Config-driven: new scenarios go in `src/opsflow/data_gen/scenarios.py`, not core logic
